# 辅导员系统部署与服务器验证文档

## 1. 当前部署架构

当前版本前端不再直接请求大模型 `af-api`，而是只访问本系统后端：

```text
浏览器
  -> http://服务器IP/counselor/
  -> OpenResty 静态资源
  -> /counselor/auth/* 代理到 Spring Boot /auth/*
  -> /counselor/api/* 代理到 Spring Boot /api/*
  -> Spring Boot 后端校验 JWT、维护用户/会话/消息
  -> Spring Boot 后端再请求 AI 服务 AGENT_BASE_URL
```

前端访问入口：

```text
http://服务器IP/counselor/
```

前端当前接口前缀：

```text
/counselor/auth/login
/counselor/auth/logout
/counselor/api/user/me
/counselor/api/conversations
/counselor/api/messages
/counselor/api/chat/stream
```

关键验证点：

```text
左下角账户信息必须来自 /api/user/me，不能是静态用户。
历史会话必须来自 /api/conversations。
点击历史会话后必须请求 /api/messages?conversationId=xxx 并回显消息。
发送消息必须请求 /api/chat/stream，并由后端保存 user/assistant 两类消息。
```

## 2. 前端环境变量

当前 `.env` 应保持：

```bash
VITE_API_BASE_URL=/counselor/api
VITE_AUTH_URL=/counselor/auth/login
VITE_AUTH_LOGOUT_URL=/counselor/auth/logout
VITE_AUTH_CALLBACK_URL=/counselor/?auth_callback=1
VITE_AUTH_AUTO_REDIRECT=true
VITE_AGENT_ID=2036332866958286848
```

`vite.config.js` 当前部署路径：

```js
base: '/counselor/'
```

## 3. 构建前端部署包

在本机项目根目录执行：

```bash
npm run build
tar -czf hfut-ai-counselor-ui-dist.tar.gz -C dist .
```

部署包解压后应直接包含：

```text
index.html
assets/
```

## 4. 后端数据库初始化

在 MySQL 服务器执行：

```bash
mysql -u root -p < backend/src/main/resources/schema.sql
```

会创建：

```text
ai_counselor
counselor_user
counselor_conversation
counselor_message
```

## 5. 后端服务配置

后端是 Spring Boot 服务，默认监听 `8080`。生产环境建议至少配置：

```bash
export SERVER_PORT=8080
export DB_URL='jdbc:mysql://127.0.0.1:3306/ai_counselor?useUnicode=true&characterEncoding=utf8&useSSL=false&serverTimezone=Asia/Shanghai&allowPublicKeyRetrieval=true'
export DB_USERNAME='root'
export DB_PASSWORD='你的数据库密码'
export JWT_SECRET='请替换为至少32字节的随机密钥'
export FRONTEND_URL='http://服务器IP/counselor/'
export AGENT_BASE_URL='http://10.66.6.226:9022/af-api'
export AGENT_ID='2036332866958286848'
export AGENT_TIMEOUT_SECONDS=300
```

如果使用郑大 CAS：

```bash
export PORTAL_CAS_BASE_URL='https://authx-service.s.zzu.edu.cn/cas'
export PORTAL_REDIRECT_URI='http://服务器IP/counselor/auth/callback'
```

如果暂时只做服务器链路验证，且不接入 CAS，可先用：

```text
POST /counselor/auth/dev-login
```

获取测试 token。

## 6. 启动后端

在服务器后端目录构建：

```bash
mvn clean package -DskipTests
```

启动：

```bash
java -jar target/ai-counselor-backend-1.0.0.jar
```

```bash
curl -i 'http://127.0.0.1:8080/health'
```

## 8. OpenResty 配置

备份：

```bash
cp /iflytek/deploy/base/nginx/nginx.conf /iflytek/deploy/base/nginx/nginx.conf.bak.$(date +%Y%m%d%H%M%S)
```

在第一个 `server { listen 80; ... }` 中，并且放在原有 `location /` 前面，加入：

```nginx
location /counselor/api/ {
    proxy_pass http://127.0.0.1:8080/api/;
    proxy_http_version 1.1;

    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

    proxy_buffering off;
    proxy_cache off;
    proxy_read_timeout 3600s;
}

location /counselor/auth/ {
    proxy_pass http://127.0.0.1:8080/auth/;
    proxy_http_version 1.1;

    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
}

location /counselor/ {
    alias /home/web/hfut-ai-counselor-ui/;
    index index.html;
    try_files $uri $uri/ /counselor/index.html;
}
```

如果后端部署在别的机器，把 `127.0.0.1:8080` 改成实际后端地址。

检查并重载：

```bash
docker exec iflyssenginx /usr/local/openresty/nginx/sbin/nginx -t
docker exec iflyssenginx /usr/local/openresty/nginx/sbin/nginx -s reload
```

## 9. 服务器 curl 验证

### 9.1 静态页面

```bash
curl -I 'http://127.0.0.1/counselor/'
```

应返回：

```text
HTTP/1.1 200 OK
```

检查构建资源：

```bash
curl -s 'http://127.0.0.1/counselor/' | grep -E 'assets/.+\.js|assets/.+\.css'
```

应能看到 `/counselor/assets/...`。

### 9.2 未登录边界

```bash
curl -i 'http://127.0.0.1/counselor/api/user/me'
```

应返回 `401` 或认证失败响应。这个结果是正确的，说明 `/api/**` 没有裸奔。

### 9.3 开发登录获取 token

仅用于服务器联调：

```bash
TOKEN=$(curl -s -X POST 'http://127.0.0.1/counselor/auth/dev-login' \
  -H 'Content-Type: application/json' \
  -d '{"username":"server_test","displayName":"服务器测试用户"}' \
  | sed -n 's/.*"token":"\([^"]*\)".*/\1/p')

echo "$TOKEN"
```

如果输出为空，检查后端日志、数据库连接和 OpenResty `/counselor/auth/` 代理。

### 9.4 当前账户信息

```bash
curl -s 'http://127.0.0.1/counselor/api/user/me' \
  -H "Authorization: Bearer $TOKEN"
```

应返回当前用户，例如：

```json
{"code":200,"message":"success","data":{"username":"server_test","displayName":"服务器测试用户"}}
```

前端左下角账户信息应显示这里返回的 `displayName` 或 `username`。

### 9.5 创建会话

```bash
CONV_ID=$(curl -s -X POST 'http://127.0.0.1/counselor/api/conversations' \
  -H "Authorization: Bearer $TOKEN" \
  -H 'Content-Type: application/json' \
  -d '{"title":"服务器验证会话"}' \
  | sed -n 's/.*"id":"\([^"]*\)".*/\1/p')

echo "$CONV_ID"
```

应输出会话 ID。

### 9.6 历史会话列表

```bash
curl -s 'http://127.0.0.1/counselor/api/conversations' \
  -H "Authorization: Bearer $TOKEN"
```

应包含刚创建的 `服务器验证会话`。前端左侧历史记录应来自该接口。

### 9.7 流式对话

```bash
curl -N -X POST 'http://127.0.0.1/counselor/api/chat/stream' \
  -H "Authorization: Bearer $TOKEN" \
  -H 'Content-Type: application/json' \
  -H 'Accept: text/event-stream' \
  --max-time 60 \
  -d "{\"conversationId\":\"$CONV_ID\",\"content\":\"你好，请做一次服务器验证。\",\"messages\":[{\"role\":\"user\",\"content\":\"你好，请做一次服务器验证。\"}],\"agentId\":\"2036332866958286848\"}"
```

正常会看到流式输出：

```text
event:message
data:{"content":"...","is_end":false}

event:done
data:{"is_end":true,"conversationId":"...","messageId":"..."}
```

### 9.8 消息历史闭环

```bash
curl -s "http://127.0.0.1/counselor/api/messages?conversationId=$CONV_ID" \
  -H "Authorization: Bearer $TOKEN"
```

应至少包含两条消息：

```text
role=user
role=assistant
```

这一步通过后，说明“发送对话 -> 后端保存 -> 历史会话点击回显”链路闭环。

## 10. 浏览器验证清单

浏览器访问：

```text
http://服务器IP/counselor/
```

按顺序检查：

1. 未登录访问时跳转学校认证，或进入登录页。
2. 登录后左下角账户信息显示真实姓名/账号，不是固定静态文案。
3. 点击“开启新会话”，左侧历史记录新增会话。
4. 发送一条消息，页面能流式显示 AI 回复。
5. 刷新浏览器，历史会话仍存在。
6. 点击历史会话，历史消息能回显。
7. 退出登录后，再访问 `/counselor/api/user/me` 返回未认证。

浏览器开发者工具 Network 中重点看：

```text
GET  /counselor/api/user/me
GET  /counselor/api/conversations
GET  /counselor/api/messages?conversationId=...
POST /counselor/api/chat/stream
```

不应再看到前端直接请求：

```text
/counselor/af-api/feign/chat/completions/...
```

## 11. 常见问题

### 11.1 页面打开是旧系统

`location /counselor/` 必须放在原有 `location /` 前面，然后重载 OpenResty。

### 11.2 页面空白

检查：

```bash
ls /iflytek/deploy/web/hfut-ai-counselor-ui
curl -s 'http://127.0.0.1/counselor/' | head
```

确认 `index.html` 和 `assets/` 存在。

### 11.3 登录后左下角还是静态用户

检查：

```bash
curl -s 'http://127.0.0.1/counselor/api/user/me' -H "Authorization: Bearer $TOKEN"
```

如果这里返回的用户不对，问题在后端认证映射或用户表；如果这里正确但页面不对，清浏览器缓存后重新访问。

### 11.4 历史会话为空

检查：

```bash
curl -s 'http://127.0.0.1/counselor/api/conversations' -H "Authorization: Bearer $TOKEN"
```

如果接口有数据但页面为空，看浏览器 Network 是否请求到了 `/counselor/api/conversations`。

### 11.5 对话失败

先检查后端是否能访问 AI：

```bash
curl -N -X POST 'http://10.66.6.226:9022/af-api/feign/chat/completions/2036332866958286848' \
  -H 'Content-Type: application/json' \
  -H 'Accept: text/event-stream' \
  --max-time 30 \
  -d '{"messages":[{"role":"user","content":"你好"}]}'
```

如果该命令不通，是服务器到 AI 服务网络或 AI 服务本身问题。

再检查本系统代理：

```bash
curl -N -X POST 'http://127.0.0.1/counselor/api/chat/stream' \
  -H "Authorization: Bearer $TOKEN" \
  -H 'Content-Type: application/json' \
  -H 'Accept: text/event-stream' \
  --max-time 60 \
  -d "{\"conversationId\":\"$CONV_ID\",\"content\":\"你好\",\"messages\":[{\"role\":\"user\",\"content\":\"你好\"}]}"
```
